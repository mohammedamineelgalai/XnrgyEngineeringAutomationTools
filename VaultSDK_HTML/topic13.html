Glossary Item Box


Just about every type of Vault customization needs to talk to the Vault server at some point.  So these concepts need to be understood first before any other types of customization are attempted.


Availability
This feature is available in all versions of Vault.  However the services available are different for each product and certain functions may not work depending on product level.


Web Services Overview
All communication with the Vault Server is done through Web Services.  Web Services is a cross-platform standard for allowing clients to make server requests.  Under the hood it's all XML data being transferred over HTTP.  Each Web Service call is similar to a function call.  Data is sent to the server, the server process data and data is returned to the client.

The Vault implementation consists of multiple services, broken down by functionality.  Each function call is autonomous.  In other words, the server doesn't maintain state information in between calls.  Function calls are also transacted which means that an edit call will either completely succeed or completely fail.  There should be no cases of an error leading to a half-completed state.  Lastly, almost all the function calls are secure.  When the call is made, authentication data is also transferred

WSE 3.0 is an additional framework for transferring binary data.  Since web services are XML, and XML is text, it makes it hard to transfer binary CAD files.  The file would need to be converted to text, transferred to the other computer and decoded.  This slows things down and increases the amount of data transfer.  WSE 3.0 fixes things by keeping binary data in a binary format, no text conversion.  This framework requires some extra steps for file transfers, but they are necessary for good performance.

Getting Started
To connect to the Vault Server you need the following things:
  • The location of the filestore server.  Usually the user is prompted for this information or it's in a configuration file.  Hard-coding a server location is a bad idea outside of testing.
  • The Vault name.  Again, this information is usually supplied by the user or a configuration file.  Some features apply to all Vaults, for example Users and Groups.  In these rare cases, the Vault name is not needed.
  • Authentication information.  This is usually done with a Vault user name and password.  If Active Directory authentication is configured, then the user's Windows credentials can be used instead.
The next step is to authenticate to the server and create the services you want to work with.  Both these steps are taken care of in the WebServiceManager or the Login dialog.

Using Service Objects
The mechanics of creating the HTTTP connection and converting the XML is all hidden within the Vault API's Service objects.    The WebServiceManager will take care of the object construction for you, so the next step is to figure out what you want to do and which services you need to use.  Navigate to this page for a list of all the services and what they are used for.
