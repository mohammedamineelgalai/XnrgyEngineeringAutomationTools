Glossary Item Box

Vault is basically a client/server framework.


Server Components
No parts of the server are customizable, and there are rules for how to get and set server data.

Autodesk Data Management Server (ADMS)
Also known as the Vault Server, this component contains most of the business logic for the Vault system.  It is the middle tier between the storage layer (database and file store) and the clients.
ADMS uses web services to expose an API for communicating with the server.  This is the only mechanism for Vault clients to communicate with the server.  The WebServiceManager or Connection object is the recommended mechanism for making web service calls to ADMS.

File Store
This component is a folder hierarchy that stores all the files uploaded to Vault.  Direct access or manipulation is not allowed.

Database
This component stores all of the non-file data.  Direct access or manipulation is not allowed.


Client Components
A Vault client is basically anything that communicates with ADMS through the web services API.  There are various pieces of client-side business logic that are spread out among the various client applications.

Vault Explorer
This application is the main mechanism for browsing and managing Vault data.  Technically there are 4 types of Vault Explorers, 1 for each product type.  So there is a "Vault Basic", "Vault Workgroup", "Vault Collaboration" and "Vault Professional" versions of Vault Explorer.  However it is usually more convenient to refer to them all as Vault Explorer.
Customization options:  Add commands, add tab views, subscribe to command events.  Click here for more information.

Autoloader
This utility is a bulk loader for CAD data.  Like all other Vault clients, it uses web service calls to send data to the server. 
This application cannot be customized.

CAD Applications
Examples of these applications include AutoCAD, Inventor and Civil 3D.   It's basically the CAD application with a plug-in that interfaces with the Vault Server.  Not only do these plug-ins send the files to the server, they also send the corresponding meta-data for the files. 
Although the CAD applications themselves can usually be customized, the Vault plug-ins are not customizable.

Job Processor
This utility takes Jobs off the queue and executes them.  Although it is technically not a Windows service, it operates like one.  It is designed to always be running with little to no user interaction.
Customization options:  Add plug-ins which contain custom job handlers.  Click here for more information.

Webserver
This component is the thing that generates the web pages for the web feature in Vault.  Although a browser views this as a server, it is viewed as a client by ADMS. 
This component cannot be customized.

Custom Applications
The web service API is publicly available, which means that anyone can write a Vault client.  In theory, a custom application can do anything that a Vault client can do since they both use the same API to communicate to ADMS.  Click here for more information.

Global Customization
It's possible to update the client-side business logic for all clients.  Event handlers can be written for certain operations in the Web Service API.   These events will fire from all clients.  For example, if you write a handler for the File CheckIn event, your handler will get notification whenever AutoCAD checks in a file and whenever Vault Explorer checks in a file.  Click here for more information.
