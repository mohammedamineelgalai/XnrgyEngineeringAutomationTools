Getting Started > How to Create Custom Jobs

Glossary Item Box

Jobserver is a nice feature that lets you distribute work across the network.  Instead of performing a task directly, a job can be queued up so that the work can be done elsewhere.  Vault comes with a few built-in job types, but the real power comes in creating your own.


Availability
This feature is not available in base Vault.  Vault Workgroup, Vault Collaboration or Vault Professional is required.


Creating Your Custom Job Type
This part is just about deciding how you want things to work.  Once you have decided what you want your job to do, you can work out some of the specifics.
What you need...
  • A name for the job type.  This just has to be a string.  We recommend starting with your company name to prevent any possible collisions. 
    Example name: Autodesk.TestJob
  • The parameters for the job.  This is just a collection of key-value strings.  What goes into this depends on what you want to do.  You might not need any parameters at all.
Now you are ready to code.


Queuing Your Job
There are two ways to get your Job on the queue.  You can add it yourself using AddJob or you can use the Lifecycle Event Editor utility to have your jobs fire automatically when things change lifecycle state.

If you are adding the job yourself you will need to create the application or plug-in that connects to the Vault Server and calls AddJob.  See the AddJob documentation for more information.

If you are adding the job through lifecycle state changes, then you don't get to decide the parameters.  See the Lifecycle Event Editor documentation for more information.


Handling Your Job
Now that you have code to get the job on the queue, you need to write the code to execute the job when it goes off the queue.  The best thing to do is write a Job Processor extension.  JobProcessor.exe will handle things like managing the queue and logging errors.  All you need to worry about is the actual Job itself. 

The first thing is to set up your project in Visual Studio or other IDE.  You want to create a library project (DLL) and it has to use the .NET framework. 

The next step is to create a public class which implements the IJobHandler interface.  This step requires you to fill in the logic for a bunch of functions.  See the IJobHandler documentation for more details on the functions you need to implement.

There are  5 assembly attributes you need to have in your code:
  1. AssemblyCompany - The name of your company. 
  2. AssemblyProduct - The name of the product. 
  3. AssemblyDescription - A description of your product.
  4. ExtensionId - A unique GUID for your product.  Visual Studio has a tool that can generate one for you.  Do not copy a GUID from another extension.
  5. ApiVersion - A string indicating which API schema your product works with.  Enter in "19.0" to work with Vault 2026.
Items 1-3 are provided by Visual Studio in the AssemblyInfo file.  You just need to check that they have accurate data.  Items 4 and 5 need to be created by you.

Once you have everything coded and built, it's time to deploy it.  To deploy you create a folder for your extension under %ProgramData%/Autodesk/Vault 2026/Extensions/.  You can name your folder whatever you want, but it's probably best to name it the same as your product name.  For example, if you DLL is called JobProcessorApiSamples, you would deploy to a folder called "%ProgramData%/Autodesk/Vault 2026/Extensions/JobProcessorApiSamples/".
You need to deploy your DLL, any referenced DLLs and any other files you may need to live alongside the DLL.  You also need a .vcet.config file to go in your folder (see below).

NOTE: The %programData% variable may not be set for all operating systems.  In Windows XP and 2003, this is usually "C:\Documents and Settings\All Users\Application Data".  In Vista and above, this is usually "C:\ProgramData".

NOTE:  You should not deploy Autodesk.Connectivity.JobProcessor.Extensibility.dll, Autodesk.Connectivity.WebServices.dll or Autodesk.Connectivity.Extensibility.Framework.dll since they are already part of the Job Processor runtime.

Once everything is deployed, you can start (or restart) JobProcessor to see your handler in action.


The .vcet.config File
This files tells the Vault extensions framework how to load your extension.  It's an XML file with a specific syntax.  Here is what the file should look like for a Job Handler extenstion.
<configuration>
  <connectivity.ExtensionSettings3>
     <extension
      interface="Autodesk.Connectivity.JobProcessor.Extensibility.IJobHandler, Autodesk.Connectivity.JobProcessor.Extensibility, Version=31.0.0.0, Culture=neutral, PublicKeyToken=215b0879a7566597"
      type="MyNamespace.MyClassName, MyAssemblyName">
      <setting key="JobType1" value="MyJobType"/>
    </extension>
  </connectivity.ExtensionSettings3>
</configuration>
You should replace "MyNamespace.MyClassName" with the full class name if the IJobHandler implementation class.  And you should replace "MyAssemblyName" with the assembly name of your DLL.  Replace "MyJobType" with the job type that your extension implements.  If you support multiple job types, add additional <setting> elements with keys that follow the pattern "JobType2", "JobType3" and so on.
