In Vault 2017, the SDK's underlying implemenation now leverages the Microsoft's WCF when communicating with the Vault server. The server moved to the WCF framework several releases ago, but Vault clients and apps using the SDK will migrate with the 2017 release. In previous releases, this communication was done using Microsoft WSE, which has been deprecated for a number of years. The underyling framework that the Vault SDK uses is considered an implemenation detail that should not affect external code, namely the consumers of our SDK. However, this encapsulation was not quite complete in the releases prior to our migration to WCF. Because of this, developers using the Vault SDK should be aware of some changes in behavior that will result from moving to the 2017 release and to WCF.

Firstly, a new DLL has been added to the Vault SDK - Autodesk.Connectivity.WebServices.WCF.dll. Consumers of the Vault SDK that deploy with the Autodesk.Connectivity.WebServices.dll must also deploy the Autodesk.Connectivity.WebServices.WCF.dll alongside it. Note that if you are developing a Vault or Job Processor extension, you do not need to deploy the WebServices DLL to the extensions directory, so you are not required to deploy the WCF either. This will mainly impact standalone Vault SDK applications.

The other expected impact due to this change is how errors like Vault exceptions and communications exceptions are handled. Previous versions of the SDK used propagated SoapExceptions and WebException types when returning an error from a web service call. These behaviors were specific to the WSE implementation, however. To make the SDK less coupled with the implementation, it will now wrap exceptions propagated from the server in the new VaultServiceErrorException and VaultServiceCommunicationException types. The intent is that these new classes will contain the same error information, but they will allow consumers of the SDK to handle exception types that are independent of the underlying framework being used by the SDK. Currently, the VaultServiceErrorException class inherits from SoapException to minimalize impact on any existing clients that have legacy error handling code. Please note that the intent that this will change in a future release, however, so it is best to update any error handling code as soon as possible.
