Here are a few guidelines to help you develop stable and performant Vault applications.

Communicating with the Vault Server
  • Make sure to handle errors properly when making a server call.  At the bare, minimum you should handle generic Exceptions.  However it's a better user experience if you provide more detailed information about why the error occurred.
    Errors from a Vault server call generally fall into one of 3 categories:
    • Error from the server.  These errors show up as SoapExceptions.
    • Error from a Web Service Command Extension.  These errors show up as ExtensionExceptions with restriction information.
    • Error from somewhere else, such as a network failure.  These can be handled as regular Exceptions.
  • Remember that Vault is a multi-user environement.  Another user might delete the file your program is currently working on.  The general rule is to expect the best but prepare for the worst.
  • Each server call is an expensive operation, so try to minimize the number of calls.
    • Don't make fequent calls.  For example, it would be bad if a Window's OnPaint event resulted in an Vault server call.
    • Try to get all the information in a single call if possible.  For example if you want to get 10 Folder objects, its better to call GetFoldersByIds once than to call GetFolderById 10 times.
      However, there is a 50 MB limit on data that can be transfered in a single call.  So if you have thousands of things to get, it might not be possible to do in a single call.
    • Cache data whenever possible.  For example, if you call GetFolderRoot once in your application, you probably don't need to call it again.
  • The lifetime of object ID values are not guaranteed.  So if you want to store data long-term, it's better to use something like a name instead of the ID.
  • If you SignIn to Vault, make sure to SignOut when you are done.  The SignOut is what frees up a license in Vault Workgroup and Professional.
  • If you did not SignIn (in other words, a login context was passed in to your code) do not SignOut.  Whoever passed you the login context may still be making server calls. 

Extending Job Processor
  • Don't have any UI in your job handler.  Job Processor is designed to be run as a service.  If you pop up a Dialog, there may not be a user around to click on it.  Effectively, the entire process would be locked up.
  • Don't assume jobs will be processed in any order.  Because mutliple Job Processors may be running on the network, you can't assume that the first job on the queue will get completed first.
  • Remember that Jobs are not executed instantly.  If the queue is backed up, it may be hours before a job reaches a job handler.
    • Don't assume that the Vault is in the same state as when the job was queued.  For example, the job may be referencing a file that no longer exists.

Extending Web Service Commands
  • Don't have any UI in your event handler.  You don't know which applications your handler may be loaded in.  It may be the case that you are loaded into a service, or console application.
  • If you want to block the operation, make sure to provide an adequate description of the problem when the Restriction is added.
  • Make sure to handle errors in your event handlers.  Don't let the Exception bubble up if it can be avoided.
  • Within your event handler, try not to perform an operation that could itself fire an event.  For example, checking out a file in your lifecycle state change event handler.
    If you can't avoid the operation, make sure to add code to prevent infinite recursion.
