Glossary Item Box

Introduction

Editing Items in the Item Service is similar to editing Files in the Document Service.  The words are different but the same three steps are involved.  Step 1:  Reserve the object for editing.  Step 2: Perform the edit.  Step 3: Finalize the changes and relinquish control of the object.

Here is a table of how those three steps work for Files and Items:

  Files (Document Service) Items (Item Service)
Reserve the object Check-out the File Create an editable Item object
Perform the Edit Edit the File on disk Modify the Item object
Finalize the changes and relinquish control Check-in the File Commit the Item

 

Creating Editable Items

Edits can only occur on editable version of Items.  An editable Item is similar to a checked-out File.  The Item is now locked to a specific user.  Nobody else can make changes.  Unfortunately there is no property on an Item object which tells if it is editable or not.  Editable Items are created by calling certain API functions.  So it's up to the client programmer to keep track of which Items are editable and which are not.

Items Locks

When an editable Item is created, an invisible lock is created on that Item.  The lock is set to a specific user and login time.  So if a client begins and edit and logs out before the edit is complete, then they will not be able complete the edit since the login time will be different.  The result is an Item which nobody can edit.  Locks expire after 20 minutes as a safeguard.  This feature also means that all Item edits must be done within a 20 minute window.  If the user begins an edit, goes to lunch, then attempts to complete the edit, the server will throw an error. 

There are 3 ways to get rid of an Item lock:
- Commit the edit.
- Undo the edit.
- Wait 20 minutes for the lock to expire.

Committing Item Changes

A commit of an Item is the same as a check-in of a File.  It finalizes the changes and removes the lock on the Item.  There are several 'commit' functions.  UpdateAndCommitItems is used in most cases, but there are cases where the 'commit' function is paired with a specific 'edit' function.  For example, RollbackItemLifeCycleStates begins the edit and CommitRollbackItemLifeCycleStates commits the edit.  There are other cases where the edit and commit is completed in a single call (ex. UpdateItemRevisionNumbers).  Make sure to read the product documentation closely.

Undoing Item Changes

It's important to roll-back the change if anything goes wrong.  If things are not cleaned up properly than the Items may be stuck in a locked state, where everybody has to wait 20 minutes before using the Item.  A lot of burden is placed on the client to keep-track-of and clean-up locked Items.

Here is the recommended code structure:

try 
{
  // get the editable item
    
    // edit the item
    
    // commit the edit
}
catch
{
       // undo the edit
}

Just like with 'commit' functions, there are several 'undo' functions.  Usually UndoEditItems is used, but in some cases the undo is paired with a specific 'edit' function.  For example, UndoRollbackItemLifeCycleStates undoes the edit started by RollbackItemLifeCycleStates.
