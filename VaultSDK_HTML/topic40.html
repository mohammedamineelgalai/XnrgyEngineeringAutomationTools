Introduction

All client/server communication, including file transfers, is done through HTTP.  The web service specification does not include transfer of binary data, so Web Service Extensions 3.0 (WSE) is used to upload and download files.   

AcquireFiles is the recommended function for downloading and checking out files.  AddFile and CheckinFile are the recommended functions for uploading files.  These functions also provide extra functionality automatically, including batch downloading, large file transfer, compression and file reference fixing.

Configuring Client Applications

4 MB is the default HTTP transfer limit allowed by WSE.  Increasing this value requires editing the client's config file.  Config files have special meaning to .NET applications.  This page just covers how to modify these files to adjust the transfer limit.  More detailed information on config files can be found in the Microsoft documentation.

The below settings need to be present in the .config file to allow a 50 MB limit.

<configuration>
      <configSections>
              <section name="microsoft.web.services3" 
         type="Microsoft.Web.Services3.Configuration.WebServicesConfiguration, Microsoft.Web.Services3, Version=3.0.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35" />
     </configSections>
     <microsoft.web.services3> 
            <messaging>
                   <maxRequestLength value="51200" />
            </messaging>
          <mtom clientMode="On" />
      </microsoft.web.services3> 
</configuration>

MTOM - WSE 3.0 uses MTOM to transfer binary data.  If MTOM is not enabled or WSE 3.0 is not installed, file transfers will still work, but they will be base 64 encoded.  For performance reasons, WSE should always be used and MTOM should always be enabled.  Base 64 encoding hurts performances since it results in more data being transferred. 

To enable MTOM, make sure that the section <mtom clientMode="On" /> is in the <microsoft.web.services3> section of the config file.

Standalone applications - In Visual Studio, create an XML file called app.config if one does not already exist.  Next copy the above XML into app.config.  If the project already has an app.config, then the sections above need to be merged into the existing file. 

When the project gets built, the app.config file gets renamed to [EXE name].config.  For example, if your application is called VaultList.exe, then VaultList.exe.config is the name of the config file.  The config file must always be in the same directory as the EXE.   When distributing the application, the config file must be distributed as well.

Application Libraries / Plug-ins - Config files are associated with EXEs not DLLs.  So if an application is a DLL it will not have a config file of its own.  In this case, a config file is required on the EXE which is running the plug-in.  Even if the EXE is not a .NET application, it can still have a config file.  For example, Inventor.exe.config is used for the Vault plug-in.

If a DLL is to be used by multiple programs, than each one needs its own config file.  If a config file exists, than the needed settings need to be merged in.

Web applications - Web applications also have a config file, which is named web.config.  If web.config does not exist, add it to the project and copy in the XML data above. If one already exists, then the XML data needs to be merged in.
